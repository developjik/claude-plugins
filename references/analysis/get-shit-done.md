# Get Shit Done (GSD) - 심층 분석

## 1. 철학과 원칙

### 핵심 철학: "Solo Developer + Workflow"
GSD는 1명의 개발자와 1개의 AI 구현자(Claude 등)를 위한 시스템입니다. 기업 수준의 복잡성(스프린트, 스토리 포인트, 이해관계자 동의)을 제거하고, 실제 필요한 것에만 집중합니다.

**"The complexity is in the system, not in your workflow"**
— 복잡성은 시스템이 담당하고, 사용자는 단순한 인터페이스만 다루면 된다는 철학.

### 5가지 핵심 원칙

1. **Fresh Context Per Agent** — 각 에이전트는 깨끗한 200K 컨텍스트 윈도우를 받음. 컨텍스트 부팅(Context Rot)을 근본적으로 해결
2. **Thin Orchestrators** — 워크플로우 파일은 오케스트레이션만 담당. 무거운 작업은 에이전트에 위임
3. **File-Based State** — 모든 상태가 `.planning/`에 사람이 읽을 수 있는 파일로 저장됨
4. **Absent = Enabled** — 설정에서 누락된 키는 기본값으로 활성화. 거부가 아닌 수용이 기본
5. **Defense in Depth** — 다층 방어 시스템 (계획 검증 → 실행 → 검증 → UAT)

### 컨텍스트 부팅(Context Rot)에 대한 인식
GSD의 출발점은 **"컨텍스트 부팅이 AI 개발의 가장 큰 품질 킬러"**라는 관찰입니다. AI 모델이 대화 기록으로 컨텍스트 윈도우를 채우면서 발생하는 품질 저하를 근본적으로 해결하고자 합니다.

## 2. 사용자 경험과 멘탈 모델

### 사용자 프로파일
- **Solo Developer**: 팀이 없는 개인 개발자
- **Visionary/Builder**: 비전 제시자 = 사용자, 구현자 = AI
- **트러스트 기반**: AI의 자동화된 작업을 신뢰하는 사용자

### 멘탈 모델
- **"Just works" 철학**: 복잡성은 시스템 내부에, 사용자는 간단한 명령어만 사용
- **자동화 우선**: "If Claude CAN do it via CLI/API, Claude MUST do it" — 가능하면 무조건 자동화
- **안전한 실패**: 실패가 발생하면 디버그 플랜을 생성해 즉시 재실행

### UX 디자인
- 최소한의 인터랙션으로 최대의 결과
- 각 단계는 명확한 입력/출력을 가짐
- 진행 상황은 실시간으로 추적 가능
- 컨텍스트 경고 시스템 (35%/25% 임계값에서 사용자에게 알림)

## 3. 워크플로우 방법론

### 주요 워크플로우
```
NEW PROJECT
   ↓
/gsd:new-project → Questions → Research → Requirements → Roadmap
   ↓
FOR EACH PHASE:
   ↓
/gsd:discuss-phase → CONTEXT.md (사용자 선호도 기록)
   ↓
/gsd:plan-phase → Research → Plans (2-3 tasks/plan) → Verification
   ↓
/gsd:execute-phase → Wave 실행 (병렬/순차) → Atomic commits
   ↓
/gsd:verify-work → UAT (수동 검수)
   ↓
/gsd:ship → PR 생성 (선택적)
   ↓
/gsd:complete-milestone → 마일스톤 완료
```

### 파생 워크플로우
- **Quick Mode**: `/gsd:quick` — 전체 플래닝을 건너뛰고 즉시 실행 (간단한 작업용)
- **Gap Closure**: 검증 실패 후 자동으로 생성되는 수정 계획
- **Assumptions Mode**: 코드베이스 분석을 통한 가정 기반 워크플로우 (기존 프로젝트에 적용 시)
- **Pause/Resume**: `/gsd:pause-work` → `/gsd:resume-work`로 세션 간 작업 이어가기
- **Forensics**: `/gsd:forensics`로 실패한 워크플로우 사후 조사

### 핵심 설계 결정
- **계획당 2-3개 태스크**: 50% 컨텍스트 타겟으로 품질 유지
- **Wave Execution**: 의존성에 따라 병렬/순차 자동 결정
- **Atomic Commits**: 태스크당 하나의 커밋으로 추적성과 롤백 보장

## 4. 혁신 포인트

### 1. Fresh Context Philosophy
모든 에이전트가 깨끗한 200K 컨텍스트 윈도우를 받는다는 단순하지만 강력한 원칙. 이것이 AI 개발의 가장 큰 품질 킬러(컨텍스트 부팅)를 근본적으로 해결합니다.

### 2. Wave Execution Model
의존성 그래프를 기반으로 태스크를 웨이브로 그룹화:
- 같은 웨이브 내의 태스크는 병렬 실행
- 다른 웨이브의 태스크는 순차 실행
- 처리량은 최대화하면서 정확성은 유지

### 3. XML Plan Format
AI 실행에 최적화된 XML 구조화 포맷:
```xml
<task type="auto">
  <name>Create login endpoint</name>
  <files>src/app/api/auth/login/route.ts</files>
  <action>...</action>
  <verify>...</verify>
  <done>...</done>
</task>
```
문서가 아닌 **프롬프트**로서의 계획. AI가 직접 소비하고 실행할 수 있는 형식.

### 4. Thin Orchestrator Pattern
워크플로우 파일은 오케스트레이션만 담당:
- 에이전트를 스폰하고 결과를 수집
- 메인 컨텍스트는 가볍게 유지
- 무거운 작업은 항상 전문 에이전트에게 위임

### 5. File-Based State with Human Readability
`.planning/` 디렉토리에 모든 상태를 저장:
- 사람이 읽을 수 있는 마크다운/JSON
- 컨텍스트 리셋을 생존
- Git에 커밋 가능 (팀 가시성)
- 디버깅과 포렌식이 용이

### 6. Runtime Abstraction
6개 서로 다른 AI 코딩 런타임을 통합 지원:
- Claude Code, OpenCode, Gemini CLI, Codex, Copilot, Cursor
- 인스톨러가 런타임별 요구사항에 맞게 파일 조정
- 벤더 종속 없는 워크플로우

## 5. 강점과 약점

### 강점

**1. 컨텍스트 부팅 해결**
- Fresh Context 원칙으로 AI 품질 저하를 근본적으로 방지
- 이것만으로도 다른 프레임워크와 차별화되는 가치

**2. 병렬 실행 생산성**
- Wave Execution으로 가능한 곳에서 병렬, 필요한 곳에서 순차
- 4명 리서처 병렬, 다수 익스큐터 병렬

**3. 검증 강화**
- 계획 검증 (Plan Checker, 최대 3회 반복)
- 실행 후 자동 검증
- 사용자 수용 테스트 (UAT)
- 다층 방어로 품질 보장

**4. 상태 지속성**
- 파일 기반 상태가 컨텍스트 리셋과 세션 경계를 생존
- 언제든 작업을 중단하고 재개 가능
- 포렌식 도구로 실패 원인 분석 가능

**5. 벤더 독립성**
- 6개 런타임 지원으로 특정 AI 플랫폼에 종속되지 않음
- 마이그레이션 비용 최소화

**6. 원자적 커밋**
- 태스크당 하나의 커밋으로 추적성, 바이섹트, 롤백 보장
- AI 자동화 워크플로우에서 git 히스토리의 깔끔함 유지

### 약점

**1. 복잡성과 학습 곡선**
- 44개 명령어, 46개 워크플로우, 16개 에이전트 이해에 시간 소요
- 초기 설정이 복잡

**2. 오버엔지니어링 리스크**
- 작은 프로젝트나 간단한 작업에는 과도한 구조
- Quick Mode로 일부 완화되지만 근본적 한계

**3. 커스터마이징 한계**
- 템플릿 기반으로 제한됨
- 비표준 워크플로우에 적응 어려움

**4. 프로세스 강제**
- 특정 방식을 강제하여 개발자 자율성 제한
- "GSD 방식"이 항상 최선은 아닐 수 있음

**5. 토큰 비용**
- 병렬 에이전트 실행으로 토큰 사용량 증가
- 비용 효율성 모델(profile)로 조절 가능하지만 기본적으로 비쌈

## 6. 에이전트/스킬 전략

### 에이전트 분류 (16개)

**리서치 그룹 (병렬 4개)**:
- Stack Researcher: 기술 스택 조사
- Features Researcher: 기능 분석
- Architecture Researcher: 아키텍처 패턴 조사
- Pitfalls Researcher: 잠재적 함정 분석

**계획 그룹 (순차)**:
- Planner: 실행 가능한 계획 생성
- Plan Checker: 계획 품질 검증 (최대 3회 반복)

**실행 그룹 (병렬/순차)**:
- Executor: 실제 구현 (Wave 내 병렬, Wave 간 순차)

**검증 그룹 (순차)**:
- Verifier: 목표 대비 산출물 검사
- Auditor: 품질 감사

**복구 그룹**:
- Debugger: 체계적 이슈 진단
- Gap Closer: 검증 실패 시 자동 수정 계획

### 에이전트 스킬 주입
- 프로젝트 특화 지식을 에이전트에 주입 가능
- `agent_skills` 설정으로 프로젝트별 커스터마이징
- 스킬이 `<agent_skills>` 블록으로 동적 로드

### 핵심 설계 원칙
- **에이전트당 단일 책임**: 한 에이전트는 한 가지 역할만
- **독립적 컨텍스트**: 각 에이전트는 깨끗한 컨텍스트
- **명확한 인터페이스**: 입력/출력이 명확히 정의됨

## 7. 컨텍스트 관리 전략

### 컨텍스트 품질 곡선
```
0-30%   : PEAK (최고 품질)
30-50%  : GOOD (양호)
50-70%  : DEGRADING (품질 저하 시작)
70%+    : POOR (품질 급격히 저하)
```
이 곡선을 기반으로 **계획당 2-3개 태스크**로 50% 컨텍스트 타겟을 설정.

### 컨텍스트 전파 아키텍처
```
PROJECT.md          → 모든 에이전트 (항상 로드)
CONTEXT.md          → Researcher, Planner, Executor
PLAN.md             → Executor, Plan Checker
SUMMARY.md          → Verifier, State tracking
STATE.md            → 세션 간 메모리
todos/              → 아이디어 캡처
threads/            → 크로스 세션 작업
seeds/              → 미래 아이디어
```

### 핵심 인사이트
- **계층적 컨텍스트**: 전체 프로젝트 → 현재 단계 → 계획 → 작업으로 점진적 상세화
- **파일이 컨텍스트의 그릇**: 상태를 파일로 관리하여 컨텍스트 리셋에도 유지
- **필요시에만 로드**: 전체 컨텍스트를 항상 로드하지 않고 단계별로 필요한 것만

## 8. 안전과 통제

### 다층 방어 시스템

**1. 사전 검증**
- Plan Checker: 계획 품질 검증 (최대 3회 반복)
- XML 구조 검증: 계획 포맷 준수 확인

**2. 실행 중 통제**
- Atomic Commits: 태스크당 커밋으로 언제든 롤백 가능
- Deviation Rules: 실행 중 예기치 않은 작업 처리 규칙
- Checkpoint System: 3가지 유형의 체크포인트

**3. 사후 검증**
- 자동화된 검증: 실행 후 목표 대비 결과 확인
- 사용자 수용 테스트 (UAT): 최종 사용자 검수
- Forensics: 실패 시 사후 조사 도구

### 보안 조치
- **프롬프트 인젝션 감지**: 사용자 입력 스캔
- **경로 순회 방지**: 파일 경로 검증
- **안전한 JSON 파싱**: 인젝션 방지
- **셸 인자 검증**: 셸 인젝션 방지

### 사용자 통제
- 각 단계마다 명시적 사용자 확인 가능
- pause/resume으로 언제든 작업 중단/재개
- Quick Mode로 프로세스 단순화 옵션

## 9. 적용 가능성

### 적합한 상황
1. **솔로 개발자의 복잡한 프로젝트**: 다단계 개발이 필요한 경우
2. **AI 기반 개발**: Claude Code, Gemini CLI 등을 적극 활용하는 개발자
3. **품질 중시**: 코드 품질과 검증이 중요한 프로젝트
4. **장기 프로젝트**: 세션 간 연속성이 중요한 경우
5. **멀티 플랫폼**: 여러 AI 코딩 도구를 사용하는 개발자
6. **기존 프로젝트 (Brownfield)**: 이미 구축된 시스템에 통합 (Assumptions Mode)

### 부적합한 상황
1. **간단한 스크립트**: 작은 자동화 스크립트에는 과도
2. **빠른 프로토타이핑**: 구조화된 프로세스가 창의성 제한
3. **팀 프로젝트**: 다인 개발에는 설계가 솔로 개발자 기준
4. **교육 목적**: 학습용으로는 복잡함
5. **저예산**: 병렬 에이전트로 토큰 비용 증가

## 10. 우리가 배울 점 (플러그인 마켓플레이스 관점)

### 1. 컨텍스트 관리의 근본적 해결
- Fresh Context 원칙은 모든 플러그인이 고려해야 할 기본 원칙
- 플러그인이 메인 컨텍스트를 오염시키지 않도록 설계해야 함
- 서브에이전트를 활용한 격리 패턴의 중요성

### 2. 파일 기반 상태의 강력함
- 데이터베이스 없이도 충분히 강력한 상태 관리 가능
- 사람이 읽을 수 있는 형태가 디버깅과 신뢰 구축에 핵심
- Git 친화적인 상태 관리가 협업을 촉진

### 3. XML Plan Format의 교훈
- 문서가 아닌 **프롬프트**로서의 계획 — AI가 직접 소비할 수 있는 형식
- 플러그인 간 데이터 교환 포맷의 표준화 필요성

### 4. Wave Execution의 확장성
- 의존성 기반 병렬 실행은 플러그인 마켓플레이스의 플러그인 실행에도 적용 가능
- 플러그인 간 의존성 관리와 실행 순서 최적화

### 5. 검증 중심 아키텍처
- Plan Checker → Executor → Verifier → UAT의 다층 검증
- 플러그인 품질 보증을 위한 다단계 검증 프로세스
- 자동 검증과 인간 검증의 적절한 조합

### 6. 런타임 추상화
- 여러 AI 코딩 플랫폼을 지원하는 추상화 계층
- 플러그인 마켓플레이스도 다양한 런타임을 지원해야 함
- 벤더 종속을 피하는 설계의 중요성

### 7. 포렌식과 복구
- 실패 시 사후 조사(forensics) 도구의 가치
- pause/resume으로 작업의 안전한 중단과 재개
- 플러그인 에러 발생 시 사용자가 원인을 파악할 수 있는 투명성

### 8. "Absent = Enabled" 철학
- 기본값이 활성화인 설계가 채택률을 높임
- 사용자가 명시적으로 비활성화하지 않으면 활성화
- 플러그인 발견성(discoverability)과 관련된 중요한 원칙
