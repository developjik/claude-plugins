# gstack - 심층 분석

## 1. 철학과 원칙

### 핵심 철학: "단일 개발자의 팀급 생산성"
Garry Tan이 직접 경험한 개발 철학으로, "AI 한 명이 예전에 스무 명이 해야 했던 일을 할 수 있는 시대"라는 전제에 기반합니다. 2026년 현재 1인 개발자가 하루 10,000~20,000줄의 코드를 생산할 수 있다는 주장.

### 세 가지 핵심 원칙 (ETHOS.md)

**1. Boil the Lake (호수 끓이기)**
- 완전한 구현을 선호하는 철학. 90% 완성도보다 100% 완성도가 70줄 더 많아도 선택
- AI의 경계 비용이 거의 제로에 가까울 때는 항상 완전한 버전을 선택
- 테스트, 오류 처리, 모든 엣지 케이스를 포함한 완전한 구현 추구
- "완벽함은 완벽할 때 끝나는 것이 아니라, 더 이상 덜어낼 것이 없을 때이다"의 역설

**2. Search Before Building (건설 전 검색)**
- 모든 것을 처음부터 만들지 말고, 이미 존재하는 해결책을 먼저 찾아라
- 세 가지 지식 계층:
  - Layer 1: 검증된 방식 (Tried and true) — 안정적이지만 한계가 있음
  - Layer 2: 새로운 유행 (New and popular) — 주의 필요
  - Layer 3: 첫 원칙 (First principles) — 가장 가치 있지만 가장 어려움
- "에ureka 순간": 기존 접근법이 왜 틀렸는지 발견하는 순간이 혁신의 시작

**3. User Sovereignty (사용자 주권)**
- AI는 추천할 뿐, 최종 결정은 항상 사용자가 내리도록
- "Iron Man suit" 철학: AI가 사용자를 보강(augment)해야지 대체(replace)해서는 안 됨
- 생성-확인 루프: AI가 추천 → 사용자 확인 → AI는 절대 확인 단계를 건너뛰지 않음
- 두 개의 AI 모델이 동의하더라도 사용자의 결정을 대체할 수 없음

## 2. 사용자 경험과 멘탈 모델

### 사용자 프로필
- **기본 사용자**: 창업가, CEO, 기술 기반의 리더
- **2차 사용자**: 처음 Claude Code를 쓰는 개발자, 테크 리드, 시니어 엔지니어
- **심리적 배경**: "AI가 내 코드를 써주길 바란다"가 아니라 "AI가 내 옆에서 일하게 하고 싶다"

### 사용자 경험 설계
1. **입문 장벽 극소화**: 30초 설치, 간단한 명령어로 바로 시작
2. **점진적 노출(Progressive Disclosure)**: `/office-hours`로 시작 → 계획 검토 → 개발 → 검토 → 배포. 복잡성을 한 번에 보여주지 않음
3. **교육과 자율성의 균형**: 철학이 preamble에 자동 주입되어 사용자가 자연스럽게 학습

### 멘탈 모델 변화
- "나는 개발자가 아니라 제품 관리자다" → "나는 CEO이자 아키텍트이자 QA 엔지니어다"
- AI 도구를 "도움"이 아닌 **"팀 구성원"**으로 인식하는 패러다임 전환
- 작은 단위가 아닌 완전한 생애 주기를 경험하며 프로세스를 자연스럽게 학습

## 3. 워크플로우 방법론

### 스프린트 구조: Think → Plan → Build → Review → Test → Ship → Reflect

**워크플로우 특징**:
- 각 스킬의 출력이 다음 스킬의 입력이 되도록 설계 (파이프라인)
- 문서 연쇄: `/office-hours` → `/plan-ceo-review` → `/plan-eng-review`
- 피드백 루프: `/qa` 발견 → 고치기 → 재검증의 강력한 사이클
- 자동화된 문서: `/ship` → `/document-release` → 모든 문자 업데이트

### 병렬 스프린트 지원
- 10~15개의 동시 스프린트 실행 가능
- 각 스프린트가 독립적인 워크스페이스에서 실행
- 관리 방식: "CEO가 팀을 관리하는 방식" — 중요한 결정만 확인하고 나머지는 자동 실행

### 계획 중심 접근법
- `/office-hours`로 시작 (YC 스타일 오피스 아워)
- 6가지 강제 질문으로 제품 재정의
- 가정 도전 (premise challenge)으로 비즈니스 모델 검증
- 구현 대안 제시 (narrowest wedge 추구)

## 4. 혁신 포인트

### 1. Headless 브라우저 CLI 아키텍처
- MCP 대신 CLI 선택한 이유: 성능, 안정성, 토큰 효율성
- 지속적인 상태 관리 (쿠키, localStorage, 탭 상태 유지)
- ~100ms 명령 실행 속도 (MCP는 3-5초)
- Ref 기반 요소 선택 (`@e1`, `@e2`, `@c1`) — CSS 선택자 없이 DOM에 접근. DOM을 수정하지 않고 참조하는 것이 핵심 혁신

### 2. Chrome 확장 사이드 패널
- 실시간 브라우저 제어 시각화
- CSS 인스펙터 (Chrome DevTools 수준)
- 활동 피드백 스트리밍
- 하위 에이전트 지원 (별도 탭별 세션)

### 3. 스킬 템플릿 시스템
- SKILL.md.tmpl → 자동 생성으로 코드와 문서의 동기화 보장
- 프롬프트 템플릿 + 코드 메타데이터 결합
- 호스트별(Claude Code, Codex, Gemini) 경로 지원

### 4. 다중 에이전트 검토 시스템
- `/review`가 7개 전문가 서브에이전트를 병렬 실행
- 각 전문가는 독립적인 새로운 컨텍스트로 실행 (컨텍스트 오염 방지)
- JSON 기반 발견 사항 및 지문 기반 중복 제거
- 0-10 품점 시스템으로 정량적 품질 측정

### 5. 디자인 툴 체인
- `/design-consultation`: 디자인 시스템 구축
- `/design-shotgun`: 다중 변형 생성 및 비교 보드 (A/B 테스트 자동화)
- `/design-html`: 프로덕션 품질 HTML 생성
- `/design-review`: 실시간 시각적 감사

## 5. 강점과 약점

### 강점

**1. 생산성 혁명**
- 100배 압축 비율 (보일러플레이트: 2일 → 15분)
- 10~15개 동시 프로젝트 관리 가능
- 완전한 생애 주기 자동화 (생각부터 배포까지)

**2. 체계적인 프로세스**
- 시작부터 배포까지의 명확한 단계
- 각 단계가 다음 단계를 위한 입력으로 연결 (파이프라인)
- QA → 고치기 → 재검증의 강력한 피드백 루프

**3. 실용성과 안정성**
- MCP보다 낮은 지연 시간과 토큰 사용량 (CLI 아키텍처 선택)
- 로컬 데몬 모델로 상태 유지
- 실제 브라우저와의 완벽한 통합

**4. 확장성**
- 새로운 스킬 추가가 템플릿 시스템으로 간단
- 다중 에이전트 아키텍처로 수평 확장
- 플러그인 마켓플레이스 친화적 설계

### 약점

**1. 학습 곡선**
- 초기 설정은 간단하지만 23개 스킬을 이해하는 데 시간 필요
- 프로세스를 따라가려면 철학 이해가 선행되어야 함

**2. 플랫폼 의존성**
- Bun, Node.js, Git 등 여러 도구 의존
- macOS에 특화된 기능 (쿠키 암호화는 Linux/Windows 미지원)

**3. 복잡성 관리**
- 많은 옵션과 스킬로 초보자 혼란 가능
- 프로젝트마다 CLAUDE.md 설정 필요

**4. AI 모델 의존성**
- 여러 AI 모델(Claude, Codex) 통합에 따른 복잡성
- 특정 모델 성능에 따른 품질 변동 가능성

## 6. 에이전트/스킬 전략

### 스킬 구조 분석

**역할 기반 스킬 설계**:
- **전략적 스킬**: `/office-hours`, `/plan-ceo-review`, `/plan-eng-review` — "무엇을 만들 것인가"에 집중
- **실행적 스킬**: `/review`, `/qa`, `/ship`, `/land-and-deploy` — "어떻게 만들 것인가"에 집중
- **지원적 스킬**: `/browse`, `/setup-browser-cookies`, `/learn` — 기반 인프라
- **안전 스킬**: `/careful`, `/freeze`, `/guard` — 리스크 관리

### 스킬 간 의존성
- `benefits-from` 속성으로 명시적 의존성 표시
- 템플릿 시스템을 통한 스킬 체이닝 (한 스킬의 출력 → 다음 스킬의 입력)
- 입력/출력 연결 고리가 명확히 설계됨

### 에이전트 전략
1. **주 에이전트**: 모든 스킬의 기본이 되는 Claude (지휘자)
2. **서브 에이전트**: `/review`에서 병렬 실행하는 7개 전문가 (각각 독립 컨텍스트)
3. **크로스 모델**: Codex와의 협업 (`/codex` 스킬) — 제2의 의견
4. **사이드 에이전트**: Chrome 확장의 별도 Claude 세션 (병렬 브라우저 작업)

## 7. 컨텍스트 관리 전략

### Preamble 시스템
모든 스킬 시작 시 실행되는 표준 preamble:
- 업데이트 확인
- 세션 추적 (활성 세션 수 확인)
- 기여자 모드 확인
- AskUserQuestion 형식 표준화
- 학습 파일 로드

### 상태 관리
- **프로젝트별 상태**: `~/.gstack/projects/`에 저장
- **세션별 상태**: `~/.gstack/sessions/`에 2시간 간격 정리
- **학습 내용**: JSONL 형식으로 누적 및 검색 가능

### 컨텍스트 최적화
- 템플릿 시스템으로 중복 제거
- 스킬 간 데이터 흐름 명시화
- 필요한 컨텍스트만 동적으로 로드

### 메모리 관리
- 순환 버퍼 (50,000 항목 제한)
- 비동기 디스크 플러시
- 학습 내용의 자동 정리 및 내보내기

## 8. 안전과 통제

### 사용자 통제 메커니즘

**1. User Sovereignty 원칙**
- AI 모델 2개 동의도 사용자 결정을 대체할 수 없음
- 항상 "AI가 추천 → 사용자 확인" 루프
- AskUserQuestion 형식으로 제어 유지

**2. 안전 장치 스킬 (계층적)**
- `/careful`: 파괴적 명령 경고 (rm -rf, DROP TABLE 등)
- `/freeze`: 특정 디렉토리로 편집 제한
- `/guard`: 최대 안전 모드 (careful + freeze)
- `/investigate`: 자동 freeze 디버깅 모드

**3. 실시간 모니터링**
- `/canary`: 배포 후 모니터링 루프
- `/benchmark`: 성능 회귀 감지
- 오류 자동 보고 및 추적

### 보안 설계
- 로컬호스트만 바인딩 (외부 노출 방지)
- Bearer 토큰 인증
- 쿠키 암호화 (macOS Keychain 접근 시 사용자 승인 필수)
- 셸 인젝션 방어

### AI 경계 설정
- 명령 허용 목록 (Chrome 확장)
- 프롬프트 주입 방어 (개발 중)
- XML 프레이밍 (안전한 데이터 전송)

## 9. 적용 가능성

### 적합한 상황
1. **스타트업 초기 단계**: 제품-시장 적합성 검증 단계
2. **기술 기반 창업가**: 코드 작성 능력과 제품 비전을 모두 가진 리더
3. **개인 프로젝트**: 복잡한 앱을 빠르게 구축할 때
4. **소규모 팀**: 공유된 프로세스와 표준화된 워크플로우 필요
5. **교육용**: 현대적인 개발 프로세스 학습

### 부적합한 상황
1. **대규모 팀**: 수십 명 이상의 팀에서는 프로세스가 복잡해질 수 있음
2. **비기술 창업가**: 코드 이해 없이는 활용 어려움
3. **규제 업계**: 엄격한 검증 프로세스가 필요한 환경
4. **간단한 프로젝트**: 과도한 장비가 필요한 경우
5. **Windows 환경**: Bun/Node.js 복합 의존성 문제

### 규모 조절
- **싱글 유저**: 1~5개 프로젝트 병렬 관리
- **소규모 팀**: 5~15개 프로젝트, 공유된 설정
- **조직급**: 여러 팀의 gstack 인스턴스 관리

## 10. 우리가 배울 점 (플러그인 마켓플레이스 관점)

### 1. 스킬 아키텍처 패턴
- **역할 기반 설계**: 스킬을 명확한 역할(전략/실행/지원/안전)로 분류
- **템플릿 시스템**: 코드와 문서의 자동 동기화로 유지보수 부담 최소화
- **명시적 의존성**: `benefits-from`으로 스킬 간 관계를 명확히 표현

### 2. 사용자 경험 설계
- **점진적 노출**: 복잡한 시스템을 단계적으로 학습 (23개 스킬을 한 번에 보여주지 않음)
- **자동화된 교육**: 철학이 preamble에 자동 주입되어 별도 교육 없이 학습
- **실용성 우선**: 이론보다 실제 문제 해결에 집중

### 3. 플랫폼 설계 원칙
- **컴파일 바이너리 배포**: 실행 환경 최소화 (Bun 활용)
- **로컬 데몬 모델**: 지속적인 상태 관리 (브라우저 세션 유지)
- **확장성 위한 설계**: 새로운 스킬 추가가 템플릿 기반으로 간단

### 4. 안전과 통제 균형
- **사용자 통제 유지**: AI의 자율성과 사용자 결정 사이 명확한 경계
- **계층화된 안전 장치**: 단일 장애점 방지 (careful → freeze → guard)
- **실시간 모니터링**: 문제 발생 즉시 감지 (canary, benchmark)

### 5. 생태계 구축 전략
- **오픈 소스 핵심**: MIT 라이선스로 최대한의 채택
- **문서화 표준화**: 모든 스킬의 일관된 문서 구조
- **커뮤니티 기여**: 기여자 모드와 피드백 시스템

### 6. 성능 최적화
- **지연 시간 최소화**: MCP 대신 CLI 선택의 합리적 근거
- **토큰 효율성**: 순수 텍스트 I/O로 JSON 오버헤드 제거
- **상태 지속성**: 명령 간 상태 유지로 초기화 비용 절감

### 7. 평가 시스템
- **다층 테스트**: 무료 정적 테스트부터 유료 E2E까지 3-tier
- **자동화된 검증**: 스킬 실행 전 검증
