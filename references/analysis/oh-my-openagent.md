# Oh My OpenAgent (OmO) - 심층 분석

## 1. 철학과 원칙

### 핵심 철학: "인간 개입 = 실패 신호"
oh-my-openagent의 가장 근본적인 철학은 **인간 개입을 시스템 실패로 간주**하는 것입니다. 이는 자율주행 자동차에서 운전자가 개입할 때 시스템에 문제가 있다고 보는 것과 같은 관점. 프로젝트의 목표는 AI 에이전트가 완벽하게 스스로 작업을 완료할 수 있는 시스템을 구축하는 것.

### 7가지 핵심 원칙

1. **Human Intervention is a Failure Signal**: 인간 개입은 병목 현상이자 실패 신호. 시스템이 잘 작동한다면 인간은 관찰만 하면 됨
2. **Indistinguishable Code**: AI가 작성한 코드는 시니어 엔지니어가 작성한 것과 구분되지 않아야 함. 품질의 기준은 인간 수준
3. **Token Cost vs Productivity**: 생산성이 크게 향상된다면 더 많은 토큰 사용은 정당화됨. 비용보다 가치
4. **Minimize Human Cognitive Load**: 인간은 오직 무엇을 원하는지만 말하면 됨. 나머지는 시스템이 알아서
5. **Predictable, Continuous, Delegatable**: 컴파일처럼 예측 가능하고 지속 가능하며 위임 가능한 시스템
6. **Ultrawork Manifesto**: 모든 것은 "Human Intent → Agent Execution → Verified Result" 루프로 작동
7. **Open Philosophy**: 단일 모델에 의존하지 않고 여러 모델을 오케스트레이션하는 개방적 접근법

### "락인(Lock-in)에 대한 저항"
프로젝트는 명시적으로 "We don't do lock-in here"를 선언. Claude, Kimi, GLM, GPT, Gemini 등 다양한 모델을 조합해 최적의 결과를 내는 방식을 추구. AI 생태계의 미래가 단일 제공자가 아닌 개방된 시장이 될 것이라는 비전.

## 2. 사용자 경험과 멘탈 모델

### 두 가지 주요 작업 모드

#### 1) Ultrawork Mode (게으른 개발자를 위한)
- 단순히 `ultrawork` 또는 `ulw`를 입력하면 모든 것이 자동으로 처리
- AI가 코드베이스를 탐색, 패턴 연구, 기능 구현, 진단 검증까지 수행
- 사용자는 그냥 기다리기만 하면 됨
- "Just do it" 모드로 완전한 자동화
- **경험의 본질**: 사용자는 "지시자"가 되고, AI는 "실행자"가 됨

#### 2) Prometheus Mode (정밀함을 원하는 개발자를 위한)
- Tab 키를 눌러 프롬테우스 모드 진입
- 프롬테우스가 마치 실제 엔지니어처럼 인터뷰를 통해 요구사항을 파악
- 명확한 계획 수립 후 실행
- 모든 단계가 문서화되고 검증됨
- **경험의 본질**: 사용자는 "협력자"가 되고, AI는 "파트너"가 됨

### 사용자 멘탈 모델
- **신뢰**: AI를 완전히 믿고 맡길 수 있다는 확신
- **예측 가능성**: 같은 입력에 대해 항상 일관된 결과를 기대
- **통제감**: 필요할 때 세부적인 통제가 가능한 유연성
- **속도**: 복잡한 작업도 인간의 인지적 부하 없이 빠르게 처리

## 3. 워크플로우 방법론

### 3단계 오케스트레이션 아키텍처

#### 1) 계획 계층 (Planning Layer)
- **사용자 + 프롬테우스**: 요구사항 정의 및 계획 수립 (인터뷰 기반)
- **메티스**: 계획 전 분석 — 숨겨진 의도, 모호점 파악
- **모무**: 계획 검증 — 명확성, 검증 가능성, 문서화 기준

#### 2) 실행 계층 (Execution Layer)
- **애틀라스**: 지휘자 역할, 계획을 작업으로 분배
- 할일 목록 관리, 학습 누적, 완료 검증

#### 3) 작업자 계층 (Worker Layer)
- **시시푸스 주니어**: 작업 실행자
- **오라클**: 아키텍처/디버깅 전문가 (읽기 전용)
- **익스플로어**: 코드베이스 탐색
- **라이브러리**: 문서/OSS 코드 검색
- **프론트엔드**: UI/UX 전문가

### 병렬 실행 전략
- 5+ 명의 전문가가 동시에 병렬 작업 수행
- 연구, 구현, 검증이 동시에 진행 (실제 개발팀처럼)
- 컨텍스트 창은 깨끗하게 유지, 결과는 준비될 때 사용

### Ralph Loop
자기 참조 실행 루프의 혁신:
- 100% 완료까지 지속
- 작업 중단 방지 시스템
- **실제 완료 vs. "끝났다"는 주장 구분** — AI가 "다 했다"고 말하는 것과 실제로 다 한 것의 차이를 시스템이 검증

## 4. 혁신 포인트

### 1) 다중 모델 오케스트레이션
단일 모델 제한에서 벗어나 각 모델의 강점을 전략적으로 활용:
- Claude: 오케스트레이션, 깊은 사고
- GPT: 딥 리즈닝, 아키텍처 결정
- Gemini: 비주얼/프론트엔드 작업
- 각 작업 유형에 최적화된 모델 자동 라우팅

### 2) Intent Gate (의도 게이트)
- 사용자 요청을 먼저 분류 (연구, 구현, 조사, 수정 등)
- 실제 의도를 파악한 후 적절한 에이전트 라우팅
- 단순한 프롬프트 실행이 아닌 진정한 이해 추구

### 3) Hash-Anchored Edit Tool
- 라인 단위 콘텐츠 해시(`LINE#ID`)로 안전한 편집
- stale-line 문제 완전 해결 (성공률 6.7% → 68.3% 향상)
- oh-my-pi에서 영감을 받은 독창적 솔루션
- AI가 파일을 편집할 때 가장 흔한 실패 모드를 구조적으로 해결

### 4) 스킬 임베디드 MCP 시스템
- 각 스킬이 자체 MCP 서버를 포함
- 작업별로 필요한 도구만 로드하여 컨텍스트 창 최적화
- 전역 MCP vs. 스킬-임베디드 MCP의 유연한 결합
- **핵심 인사이트**: 컨텍스트 윈도우는 소중한 자원, 불필요한 도구를 로드하지 마라

### 5) Deep Initialization (`/init-deep`)
- 계층적 AGENTS.md 파일 자동 생성
- 프로젝트 > 소스 > 컴포넌트 수준 컨텍스트 관리
- 에이전트가 관련 컨텍스트를 자동으로 로드
- **핵심 인사이트**: 컨텍스트를 한 번에 다 로드하지 말고, 계층적으로 필요한 것만

### 6) 카테고리 기반 라우팅
작업을 의미적 카테고리로 분류하여 자동 라우팅:
- `visual-engineering`: 프론트엔드, UI/UX, 디자인
- `deep`: 자율 리서치 + 실행
- `quick`: 단일 파일 변경, 오타
- `ultrabrain`: 어려운 로직, 아키텍처 결정

## 5. 강점과 약점

### 강점

**1. 완전한 자율성**
- 인간 개입 없이도 복잡한 작업 완료 가능
- Ultrawork 모드의 "Just do it" 철학
- Ralph Loop로 100% 완료 보장

**2. 모델 최적화**
- 각 작업에 최적의 모델 자동 배치
- 벤더 종속 없는 유연성
- 폴백 체인으로 안정성 보장

**3. 병렬 처리**
- 여러 에이전트 동시 작업으로 빠른 결과
- 실제 개발팀의 병렬 작업 방식 모방
- 백그라운드 에이전트로 메인 작업 방해 없음

**4. 검증 시스템**
- CI 통과, 5 에이전트 병렬 검토, 자동화 검증
- 다단계 검증으로 높은 코드 품질
- 실제 완료 vs. 주장된 완료 구분

**5. 확장성**
- 스킬 시스템으로 새로운 기능 쉽게 추가
- 카테고리 시스템으로 새로운 작업 유형 확장
- 스킬 임베디드 MCP로 도구 확장

**6. 체계적인 구조**
- 200 LOC 하드 리밋으로 모듈 크기 제한
- 단일 책임 원칙 절대 준수
- 명확한 에이전트 역할 분리

### 약점

**1. 복잡성**
- 9개 에이전트, 48개 훅, 26개 도구 이해에 시간 소요
- 초기 설정이 복잡
- 내부 작동 방식 이해가 어려움

**2. 다중 모델 의존성**
- 여러 LLM 제공자에 대한 의존성
- API 키 관리 복잡
- 특정 모델 장애 시 영향 범위 큼

**3. 비용**
- 병렬 에이전트 실행으로 토큰 사용량 증가
- 다중 모델 호출로 API 비용 증가
- 소규모 프로젝트에 비용 부담

**4. 디버깅 어려움**
- 다중 에이전트 상호작용으로 문제 원인 파악 어려움
- 너무 추상화된 경우 실제 작동 방식 이해 한계
- 에러 메시지가 구체적이지 않을 수 있음

**5. 리소스 요구사항**
- 여러 에이전트 동시 실행에 대한 리소스 요구
- 백그라운드 에이전트 관리 오버헤드

## 6. 에이전트/스킬 전략

### 11개 전문 에이전트 (그리스 신화 기반 네이밍)

**오케스트레이션 계층**:
- **Sisyphus**: 주요 오케스트레이터 (Claude Opus 4.6). 전체 워크플로우 조율
- **Atlas**: 할일 목록 지휘자. 계획을 작업으로 분배
- **Prometheus**: 전략적 계획 수립. 인터뷰 모드로 요구사항 파악

**분석/검증 계층**:
- **Metis**: 계획 분석가. 숨겨진 의도와 모호점 파악
- **Momus**: 계획 검증자. 명확성, 검증 가능성, 문서화 기준

**실행 계층**:
- **Sisyphus Junior**: 카테고리 기반 실행자. 실제 작업 수행
- **Hephaestus**: 깊은 작업 수행 (GPT-5.4). 자율 리서치와 실행
- **Frontend**: UI/UX 전문가

**지원 계층**:
- **Oracle**: 아키텍트/디버깅 전문가 (읽기 전용)
- **Library**: 문서/OSS 코드 검색
- **Explore**: 빠른 코드베이스 탐색
- **Multimodal Looker**: 비주얼 콘텐츠 전문가

### 스킬 시스템
- **git-master**: 원자적 커밋, 리베이스 수술, 히스토리 검색
- **work-with-pr**: PR 전체 생애주기 관리 (3단계 검증)
- **playwright**: 브라우저 자동화
- **frontend UI/UX**: 디자인 우선 UI 개발
- **커스텀 스킬**: `.opencode/skills/*/SKILL.md`로 사용자 정의

### 스킬-카테고리 결합 시스템
작업의 카테고리에 따라 필요한 스킬과 에이전트를 자동으로 선택. "시각적 엔지니어링" 카테고리면 프론트엔드 에이전트 + playwright 스킬이 자동 로드.

## 7. 컨텍스트 관리 전략

### 3계층 컨텍스트 주입
1. **프로젝트 레벨**: 전체 컨텍스트 AGENTS.md
2. **디렉토리 레벨**: 특정 영역 컨텍스트 AGENTS.md
3. **파일 레벨**: 세부 컨텍스트

### 컨텍스트 최적화 기법
- **Hashline 편집**: 라인 추적으로 오래된 컨텍스트 문제 해결
- **배경 에이전트**: 메인 에이전트 작업 방해하지 않는 컨텍스트 수집
- **지식 누적**: 작업 간 학습 결과 전파 (한 에이전트가 배운 것을 다른 에이전트가 활용)
- **LSP + AST-Grep**: IDE 수준의 정확한 컨텍스트 접근 (불필요한 파일 읽기 최소화)

### 컨텍스트 제한 전략
- **적극적 생략**: 불필요한 컨텍스트는 과감히 제거
- **캐싱**: 학습 결과 저장 후 재사용
- **스킬 임베디드 MCP**: 컨텍스트 창에 불필요한 도구 추가 방지

### 핵심 철학
컨텍스트 윈도우는 **소중한 자원**. 모든 도구, 모든 문서, 모든 컨텍스트를 항상 로드하는 것이 아니라, 작업에 필요한 최소한만 로드.

## 8. 안전과 통제

### 강력한 통제 메커니즘

**1. Todo Enforcer**
- 유휴 에이전트가 작업에 복귀하도록 강제
- AI가 "끝났다"고 말하며 빈둥거리는 것을 방지

**2. Comment Checker**
- AI 슬롭 코멘트 제거
- AI가 생성한 불필요한 주석 자동 감지

**3. 유휴 탐지**
- 에이전트가 "작업 완료"라고 거짓으로 말하는 것 방지
- 실제 진행 상황과 주장된 진행 상황 비교

**4. 검증 루프**
- 모든 작업이 검증될 때까지 계속 (Ralph Loop)
- "완료" 선언 전 실제 검증 수행

### 에이전트별 권한 제한
- **Oracle**: 읽기 전용으로 제한 (실수로 파일 수정 방지)
- **Library/Explore**: 쓰기/위임 금지 (검색 전용)
- **Atlas**: 다시 위임 금지 (무한 루프 방지)

### 모델 안전장치
- **폴백 체인**: 모델 실패 시 대체 모델 자동 전환
- **제공자 우선순위**: 안정적인 서비스 우선 배치
- **실험적 기능**: 보수적 기본값으로 설정

### 사용자 통제 유지
- **예측 가능성**: 같은 입력 항상 같은 결과
- **투명성**: 에이전트 의사결정 과정 추적 가능
- **개입점**: 필요한 경우 사용자 개입 가능한 지점 마련

## 9. 적용 가능성

### 적합한 상황
1. **대규모 복잡 프로젝트**: 여러 의존성과 복잡한 아키텍처
2. **장기 개발 작업**: 며칠에 걸친 개발
3. **다중 모델 환경**: 이미 여러 LLM 서비스 사용 중
4. **생산성 최우선**: 개발 속도가 가장 중요한 팀
5. **반복적인 작업**: 유사한 패턴의 여러 작업
6. **자동화된 CI/CD**: PR 검증, 배포 자동화

### 부적합한 상황
1. **단순 작업**: 빠른 수정이나 작은 변경
2. **제한된 예산**: 병렬 실행으로 토큰 비용 증가
3. **교육 목적**: 내부 작동 방식 학습에 부적합
4. **단일 모델 환경**: 다중 모델 오케스트레이션의 이점을 활용할 수 없음
5. **엄격한 보안 요구**: 다중 API 호출로 공격면 증가

## 10. 우리가 배울 점 (플러그인 마켓플레이스 관점)

### 1) 모듈 아키텍처의 엄격함
- 200 LOC 하드 리밋: 파일이 길어지면 분리해야 함
- index.ts는 엔트리포인트만, 비즈니스 로직 금지
- catch-all 파일(utils.ts, service.ts) 금지
- **적용**: 플러그인의 크기와 복잡도에 가이드라인 설정 필요

### 2) 스킬 임베디드 MCP
- 각 스킬이 자체 MCP 서버를 포함하는 구조
- 컨텍스트 윈도우 오염 최소화
- **적용**: 플러그인이 필요한 도구만 로드하는 메커니즘

### 3) 카테고리 기반 라우팅
- 작업을 의미적 카테고리로 분류하여 자동 라우팅
- **적용**: 플러그인 마켓플레이스에서 작업 유형별로 적합한 플러그인 자동 추천

### 4) 인간 개입 최소화 철학
- "인간 개입 = 실패"라는 강력한 철학
- **적용**: 플러그인이 사용자 개입을 최소화하도록 설계. 자동화가 기본

### 5) 다단계 검증
- work-with-pr 스킬의 3단계 검증 (CI + 5 에이전트 검토 + 자동화 검증)
- **적용**: 플러그인 품질 보증을 위한 다단계 검증 프로세스

### 6) 컨텍스트 관리의 세밀함
- 계층적 AGENTS.md, 스킬 임베디드 MCP, LSP/AST 활용
- **적용**: 플러그인이 메인 컨텍스트를 최소한으로 사용하도록 가이드

### 7) 에이전트 권한 관리
- Oracle은 읽기 전용, Library는 쓰기 금지 등 역할별 권한
- **적용**: 플러그인의 권한을 최소 권한 원칙으로 관리

### 8) Ralph Loop의 끝까지 가는 철학
- 100% 완료까지 지속, 중간에 포기하지 않음
- **적용**: 플러그인이 작업을 중간에 포기하지 않고 끝까지 완수하는 패턴
