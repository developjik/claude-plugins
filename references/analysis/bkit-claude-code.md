# bkit (Vibecoding Kit) - 심층 분석

## 1. 철학과 원칙

### 핵심 철학: "AI-Native Development OS"
bkit은 단순한 코딩 보조 도구가 아니라 **AI 네이티브 개발 운영체제**를 지향합니다. 세 가지 핵심 철학이 이를 관통합니다:

1. **Automation First**: 사용자가 PDCA 방법론을 몰라도 자동으로 적용되도록 설계. 사용자가 의식하지 않아도 시스템이 알아서 품질 관리 사이클을 돌림.
2. **No Guessing**: 확실하지 않을 때는 문서 확인, 문서에 없으면 사용자 질문. AI의 추측을 원천 차단.
3. **Docs = Code**: 설계 먼저, 구현 나중. 설계와 구현이 항상 동기화되도록 강제.

### 4가지 AI-Native 원칙
- **Specification-Driven Development**: 명확한 명세를 통한 정확한 코드 생성. 모호함은 버그의 근원이라는 전제.
- **Context-Aware Development**: 프로젝트 컨텍스트에 기반한 AI 의사결정. 단순한 프롬프트가 아닌 환경 인식.
- **AI Agents as Developers**: AI 에이전트의 자율적인 개발 역할 수행. 인간은 오케스트레이터.
- **Human Oversight by Design**: 점진적 신뢰를 통한 인간 통제 (Controllable AI). 자동화와 통제의 균형.

### 핵심 가치관
- **공학적 접근**: AI 개발에도 엔지니어링 엄밀성을 적용해야 한다는 신념
- **프로세스 중심**: 결과보다 프로세스를 최적화하면 결과는 따라온다는 철학
- **검증 중시**: plausible(그럴듯함) ≠ correct(정확함), 90% 일치율이라는 엄격한 문턱값
- **제약 설계**: 무한한 커스터마이제이션이 아닌 의도된 제한으로 학습성 향상

## 2. 사용자 경험과 멘탈 모델

### 사용자 여정 단계별 설계
1. **SessionStart**: CLI 대시보드로 즉시 상황 파악 (프로그레스 바, 워크플로우 맵)
2. **Level Detection**: 자동 프로젝트 수준 인식 (Starter/Dynamic/Enterprise)
3. **PDCA Auto-Apply**: 상태 머신을 통한 자동 PDCA 적용
4. **Interactive Checkpoints**: 각 단계별 사용자 확인 포인트

### 멘탈 모델 변화 전략 (Before → After)
- "어디서 시작해야 할지 모르겠다" → CLI 대시보드 + 파이프라인 가이드로 즉시 방향 제시
- "그냥 코드 먼저 짜자, 문서는 나중에" → 상태 머신이 강제로 습성 형성
- "계속 같은 실수를 반복한다" → 회귀 가드 + 메트릭 컬렉터로 반복 차단

### 레벨별 사용자 프로필
- **Starter**: 초보자, 정적 웹. 친절한 설명과 단순화된 PDCA 제공
- **Dynamic**: 중간 수준, 풀스택. 기술적이되 명확한 설명
- **Enterprise**: 전문가, 마이크로서비스. 간결하고 아키텍처 중심 접근

## 3. 워크플로우 방법론

### PDCA 상태 머신 (v2.0.0)
```
idle → PM → Plan → Design → Do → Check → Act → Report → completed
```
- 20개 상태 전이, 9개 가드 조건
- YAML 기반 워크플로우 DSL로 선언적 정의
- 3개 프리셋: default, enterprise, hotfix
- 체크포인트/롤백 시스템 내장

### 9단계 개발 파이프라인
각 단계마다 별도의 PDCA 사이클이 중첩 실행:
1. Schema/Terminology → 2. Convention → 3. Mockup → 4. API → 5. Design System → 6. UI → 7. SEO/Security → 8. Review → 9. Deployment

### 레벨별 파이프라인 적용
- **Starter**: 1→2→3→5(선택)→6(정적)→7(SEO)→9 (4단계 생략)
- **Dynamic**: 1→2→3→4→5→6→7→8→9 (전체)
- **Enterprise**: 전 단계 필수 + 추가 엔터프라이즈 검증

### Check-Act 반복 루프
gap-detector가 매치율을 분석 → 90% 미만이면 자동 개선 → 최대 5회 반복 → 여전히 부족하면 사용자에게 보고

## 4. 혁신 포인트

### Context Engineering 구현
전통적 "프롬프트 엔지니어링"을 넘어 **시스템적 컨텍스트 설계**로 진화:
- 6 레이어 훅 시스템 (hooks.json → 스킬 → 에이전트 → 트리거 → 스크립트 → 팀 오케스트레이션)
- 88개 모듈, ~620+ 함수로 구성된 컨텍스트 파이프라인
- 37개 스킬과 32개 에이전트가 유기적으로 연결

### Controllable AI (L0-L4)
자동화 수준을 5단계로 세분화:
- L0: 완전 수동 → L1: 가이드 → L2: 세미-오토(기본값) → L3: 오토 → L4: 풀-오토
- 신뢰 점수 시스템 (0-100): 6가지 가중치 구성 요소로 점진적 신뢰 구축
- 긴급 정지, 체크포인트/롤백, 파괴적 작업 감지 내장

### CTO-Led Agent Teams
- CTO 역할 에이전트가 전체 PDCA 워크플로우 조율
- 병렬 에이전트 실행 (Dynamic: 3개, Enterprise: 5개)
- PM Agent Team: 사전 기획 단계에서 43개 프레임워크 통합 분석

### Skill Evals 프레임워크
- 스킬 품질 관리 시스템으로 스킬의 생애주기 관리
- A/B 테스팅: 모델 간 성능 비교
- 스킬 분류: Workflow(영구적 가치) / Capability(모델 의존적) / Hybrid
- 모델 파리티 테스트: 스킬이 더 이상 필요하지 않을 때 감지 (모델 자체 능력 향상 시)

## 5. 강점과 약점

### 강점
1. **시스템적 설계**: 단순한 기능 모음이 아닌 통합된 아키텍처. 모든 컴포넌트가 유기적으로 연결
2. **점진적 학습**: L0-L4 자동화 수준으로 사용자가 성장함에 따라 자동화 확대
3. **검증 강화**: 90% 일치율, 7단계 품질 게이트, 회귀 가드로 품질 보장
4. **컨텍스트 엔지니어링**: 최적화된 컨텍스트 전달로 AI 성능 극대화
5. **투명성**: JSONL 감사 로그, 의사결정 추적으로 블랙박스 해소
6. **확장성**: 모듈화된 설계로 스킬/에이전트 추가 용이

### 약점
1. **복잡성**: 학습 곡선이 가파름. 88개 모듈, 37개 스킬, 32개 에이전트 이해에 시간 소요
2. **의존성**: Claude Code v2.1.78+ 필요, 에이전트 팀은 실험적 기능
3. **리소스 요구량**: 많은 상태 파일과 컨텍스트 소비. 경량 프로젝트에 부적합
4. **유연성 부족**: 엄격한 구조가 때로는 제약으로 작동
5. **성능 오버헤드**: 상태 머신 관리로 인한 오버헤드 발생 가능

## 6. 에이전트/스킬 전략

### 스킬 분류 시스템 (v2.0.6)
- **Workflow 스킬 (18개)**: 프로세스 자동화 (PDCA, 파이프라인 등). 모델과 무관한 영구적 핵심 가치
- **Capability 스킬 (18개)**: 모델 능력 증강 (모의, API 등). 모델 업그레이드 시 불필요해질 위험
- **Hybrid 스킬 (1개)**: 프로세스 + 능력 조합

이 분류의 핵심 인사이트: Workflow 스킬은 시간이 지나도 가치가 유지되지만, Capability 스킬은 모델이 발전하면 폐기될 수 있음. 이를 인식하고 관리하는 것이 장기적 건강성에 중요.

### 에이전트 아키텍처
- **모델 선택 전략**: Opus(전략적 의사결정), Sonnet(실행), Haiku(모니터링/단순 작업)
- **프론트매터 구조**: 모델, 노력 수준, 최대 턴, 메모리, 도구 제한을 선언적 정의
- **3티어 보안**: permissionMode, disallowedTools, 글로벌 설정으로 계층적 권한 관리

### 에이전트 팀 오케스트레이션
- **CTO 팀**: 아키텍처 결정, 팀 구성, 품질 강제
- **PM 팀**: 제품 발견, 전략 분석, 시장 조사
- **평행 실행**: 각 팀원이 독립적 세션에서 실행, 결과를 취합

## 7. 컨텍스트 관리 전략

### 컨텍스트 계층 구조
```
L1: Plugin Policy (bkit 기본값)
↓
L2: User Config (~/.claude/bkit/)
↓
L3: Project (.bkit/state/)
↓
L4: Session (런타임)
```
상위 레이어가 하위 레이어를 오버라이드하는 방식. 프로젝트 > 사용자 > 플러그인 기본값.

### 다중 레벨 훅 시스템
- 18개 이벤트: SessionStart, PreToolUse, PostToolUse 등
- 통합 핸들러: 중복 로직 제거, 일관성 유지
- 조건부 실행: matcher 필터링으로 이벤트 특정 시점에만 실행

### 상태 관리 시스템
- **PDCA 상태**: `.bkit/state/pdca-status.json`
- **신뢰 프로파일**: `.bkit/state/trust-profile.json`
- **메모리 저장소**: `.bkit/state/memory.json`
- **에이전트 상태**: `.bkit/runtime/agent-state.json`

### 리빙 컨텍스트 시스템 (v2.0.6)
- 7개 모듈: 로더, 불변성 검사자, 영향 분석기 등
- 자가 치유 에이전트: 자동 오류 복구
- PDCA 핸드오프 개선: 30-40% → 75-85% 컨텍스트 보존율 달성

## 8. 안전과 통제

### 5단계 자동화 통제 (L0-L4)
- **L0**: 모든 수동, 파괴적 작업 거부/질문
- **L1**: idle→pm, idle→plan 자동, 읽기 작업 자동
- **L2**: 대부분 전이 자동, 비파괴적 작업 자동 (기본값)
- **L3**: 모든 전이 자동 (보고→아카이브 제외), 대부분 자동
- **L4**: 완전 자동, 사후 검토만

### 파괴적 작업 감지 (8규칙)
- `rm -rf*`, `git push --force*`, `git reset --hard*` 등 차단
- 폭발 반경 분석: 영향 범위 계산 후 사용자 경고
- 사용자 확인: 중요 작업 전 항상 확인 프롬프트

### 품질 게이트 시스템
- 7단계 품질 게이트: PM→Plan→Design→Do→Check→Act→Report
- 10개 메트릭 (M1-M10): 계획 정확성, 설계 커버리지 등
- 회귀 가드: 이전 버전과 비교하여 품질 저하 감지

### 긴급 정지 메커니즘
- 즉시 정지: `/control stop` 명령어
- 체크포인트: 각 단계 전이 시 자동 생성
- 롤백: 이전 체크포인트 상태로 복원 가능

## 9. 적용 가능성

### 최적 사용 시나리오
- **대규모 프로젝트**: 마이크로서비스 아키텍처 등 복잡한 시스템
- **팀 협업**: 표준화된 개발 프로세스가 필요한 환경
- **엔터프라이즈**: 품질 관리, 감사 추적 요구사항
- **복잡한 기능 개발**: 다단계 개발 파이프라인이 필요한 기능
- **규제 환경**: 추적 가능성과 문서화가 중요한 산업

### 부적합 시나리오
- **단일 파일 수정**: 지나치게 복잡한 오버엔지니어링
- **비개발 작업**: 문서 작성, 연구 등 비기술적 작업
- **빠른 프로토타이핑**: 빠른 반복이 필요한 초기 탐색 단계
- **간단한 개인 프로젝트**: 과도한 구조가 부담

### 레벨별 적합성
- **Starter**: 학습용, 정적 웹사이트, 개인 블로그
- **Dynamic**: 중규모 풀스택 애플리케이션, SaaS
- **Enterprise**: 대규모 분산 시스템, 마이크로서비스

## 10. 우리가 배울 점 (플러그인 마켓플레이스 관점)

### 아키텍처 패턴
1. **계층화된 설계**: 도메인 지식 → 행동 규칙 → 상태 관리의 명확한 분리
2. **이벤트 기반 트리거**: 18개 훅 이벤트로 체계적 실행 제어
3. **상태 머신**: 선언적 상태 전이로 예측 가능한 워크플로우
4. **모듈화된 컴포넌트**: 88개 모듈로 유지보수 용이

### 제품 전략
1. **점진적 진화**: v1.0 → v2.0 단계적 발전 (기능 추가 vs 아키텍처 변경의 명확한 구분)
2. **신뢰 구축**: 신뢰 점수, 품질 게이트로 사용자의 AI에 대한 확신 구축
3. **에코시스템**: 스킬/에이전트 생태계를 활성화하는 구조
4. **커뮤니티**: 사용자 피드백 기반 개선 문화 (Plan Plus 스킬의 커뮤니티 기여 사례)

### 구체적 인사이트
1. **Workflow vs Capability 분류**: 플러그인이 모델 의존적인지 아닌지 구분하는 것이 장기적 품질 관리에 핵심
2. **자동화 수준 조절**: 단계적 자동화로 사용자가 성장함에 따라 점진적으로 제어 위임
3. **검증 프레임워크**: 플랫폼 차원에서 플러그인 품질을 측정하는 시스템의 필요성
4. **에이전트 팀 시너지**: 다중 에이전트가 단일 에이전트보다 나은 결과를 내는 구조적 조건
5. **상태 관리의 중요성**: 지속적 상태 추적이 경험 개선의 핵심

### 마켓플레이스 적용 방안
1. **플러그인 분류 체계**: Workflow/Capability/Hybrid 분류로 품질 관리 차등화
2. **레벨 기반 접근**: 사용자 수준별 적합한 플러그인 추천
3. **검증 프레임워크**: 플러그인 품질 자동 측정 시스템 (Skill Evals 참고)
4. **안전장치 표준화**: 파괴적 작업 감지, 롤백 기능을 마켓플레이스 기본 요구사항으로
5. **문서 중심 가이드라인**: 명세-구현 동기화를 플러그인 개발 가이드라인에 반영
