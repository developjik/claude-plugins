# Design Skill — PDCA 2단계

Plan 문서를 기반으로 **"코드를 어떻게 바꿀 것인가?"** 를 구체화합니다.

## 프로세스

### 1. Plan 문서 로드
$ARGUMENTS 에서 대상 기능의 Plan 문서를 찾아 읽습니다.

### 2. 현재 코드베이스 분석
- 관련 파일 구조 파악
- 기존 패턴과 컨벤션 확인
- 의존성 관계 분석

### 3. 변경 계획 수립

파일 단위로 구체적인 변경 목록을 작성합니다:

```markdown
# [기능명] 설계 문서

## 아키텍처 개요
[mermaid 다이어그램]

## 파일 변경 계획

### 생성 (Create)
| 파일 | 역할 | 핵심 내용 |
|:-----|:-----|:---------|
| `src/auth/login.ts` | 로그인 핸들러 | JWT 검증, 세션 생성 |

### 수정 (Modify)
| 파일 | 변경 사항 |
|:-----|:---------|
| `src/routes.ts` | 로그인 라우트 추가 |

### 삭제 (Delete)
| 파일 | 이유 |
|:-----|:-----|
| `src/auth/legacy.ts` | 레거시 코드 제거 |

## 기술 결정
- [결정] — 이유: [근거]

## 테스트 전략
- 단위 테스트: [대상]
- 통합 테스트: [대상]

## 구현 순서
1. [1단계]
2. [2단계]
3. [3단계]
```

## 출력

```
✅ Design 완료

📐 변경 요약:
- 생성: X개 파일
- 수정: Y개 파일
- 삭제: Z개 파일

➡️ 다음 단계: /implement 로 TDD 구현을 시작하세요.
```
